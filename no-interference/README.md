# Context 
All of the recordings have been taken while the `cyclictest` command was executing. The goal is to identify a pattern of behaviour of the task inside the kernel. This will be useful to identify the task in the presence of the noise generated by an hmi in order to identify *if a RT task is affected by this noise, thus causing it do miss the deadlines.*

```bash
cyclictest -t 1 -a 1 -p 99 -i 2000000 -q
```
Options :
- `-t 1`: number of spawned threads
- `-a 1`: affinity, core to pin the process to
- `-i 2000000` : number o microseconds, so 2 seconds
- `-q`: quiet option for cli print

> [!NOTE]
> Despite using the `-t 1` option, the `cyclictest` command generates **two threads**. Only the second thread (the one with the higher PID) performs the 2-second `nanosleep` wait.
> 
> You can verify this behaviour by running : `strace -p <thread_pid>`



> [!CAUTION]
> ### **Handling Dropped Events**
> In full kernel traces, depending on the board's available memory, you may encounter timestamp gaps marked as `EVENTS_DROPPED`.
> 
> This indicates that the kernel buffer is filling up faster than `trace-cmd` can read and save data to the disk. To resolve this, you must **increase the buffer size**.
> 
> `trace-cmd` provides the `-b` option to specify the buffer size. **Warning:** Do not set this value too high, as the maximum capacity depends on the specific hardware. A recommended starting point is `-b 100000` and try to increase it depending on the board's memory.
